MCP-Based Agentic AI Architecture for PLM Data Migration: A Comprehensive Framework Utilizing Ollama, ETL/ELT Strategies, and Context-Aware Data Transformation
Executive Summary
The modernization of Product Lifecycle Management (PLM) systems constitutes one of the most formidable challenges in contemporary enterprise data engineering. As manufacturing organizations transition from monolithic, on-premise legacy systems to agile, cloud-native platforms, they encounter the "context gap"—a fundamental disconnect between the rigid, relational schemas of the past and the semantic, object-oriented requirements of modern digital threads. Traditional data migration methodologies, predominantly reliant on static Extract-Transform-Load (ETL) scripts and manual mapping exercises, have proven insufficient for preserving the intricate web of relationships inherent in engineering data. These legacy approaches frequently result in data loss, semantic corruption, and extended project timelines, as they fail to capture the "tribal knowledge" embedded within unstructured datasets and complex Bill of Materials (BOM) hierarchies.
The emergence of Agentic AI, specifically the deployment of autonomous agents powered by Large Language Models (LLMs) running locally via Ollama, offers a transformative solution. By standardizing tool access through the Model Context Protocol (MCP) and leveraging graph-based contextual memory, organizations can deploy a "Sovereign Stack" that automates the migration process while adhering to strict data privacy mandates. This report delineates a comprehensive reference architecture for an MCP-based Agentic AI migration factory. It explores the shift from rigid ETL to adaptive Agentic ELT, the utilization of Knowledge Graphs for context preservation, and the implementation of a user-centric Automapping interface that bridges the gap between human expertise and machine reasoning using jQuery QueryBuilder and GraphQL.
1. The Crisis of Legacy PLM Data Migration
1.1 The High Cost of Context Loss
Product Lifecycle Management (PLM) systems serve as the intellectual backbone of manufacturing enterprises, managing the complete lifecycle of a product from ideation through engineering design and manufacturing to service and disposal. Over decades of operation, these systems accumulate vast repositories of data that are not merely tabular records but complex, interconnected networks of engineering intent. A single "Part" record in a legacy system like Teamcenter or Windchill is inextricably linked to CAD files, supplier specifications, regulatory compliance documents, and engineering change orders (ECOs).
The fundamental failure of traditional migration projects lies in their treatment of this data as simple rows and columns. When organizations attempt to "lift and shift" data using standard SQL-based ETL tools, they inadvertently strip away the semantic context. A relational table might link a Part ID to a Material ID, but it fails to capture the reason for that link—perhaps a specific deviation authorization granted ten years prior. When this flattened data arrives in the new system, the links are broken, and the "Digital Thread" is severed. Industry analysis suggests that fixing data issues post-migration is exponentially more expensive than addressing them in-flight, yet traditional tools lack the cognitive capability to inspect and repair data contextually during the transition.1
1.2 The Limitations of Deterministic Scripts
The prevailing methodology for PLM migration relies on deterministic scripting. Engineers write thousands of lines of code to map Source_Table_A.Column_1 to Target_Object_B.Attribute_2. This imperative approach assumes a static and well-understood source schema. However, empirical evidence from countless migration failures demonstrates that legacy PLM environments are rarely static. They are riddled with ad-hoc customizations, "ghost" fields that haven't been used in years but contain critical historical data, and inconsistent data entry practices (e.g., "Steel," "STL," and "Stl." all referring to the same material).
Deterministic scripts are brittle; they break when encountering data anomalies they were not explicitly programmed to handle. They cannot infer that a missing value in the "Weight" column can be derived from the associated CAD file metadata. This rigidity forces migration teams into endless cycles of "test-fail-fix," dragging out timelines and inflating costs. The industry requires a shift from imperative instruction ("do exactly this") to declarative goal-seeking ("migrate this assembly and resolve inconsistencies based on these rules")—a shift enabled only by Artificial Intelligence.3
1.3 The Imperative for Data Sovereignty
While Generative AI offers the cognitive capabilities to solve these mapping challenges, the nature of PLM data introduces a critical constraint: data sovereignty. Engineering data constitutes the core Intellectual Property (IP) of a manufacturing firm. It often includes unpatented designs, trade secrets, and export-controlled technical data (ITAR/EAR). Uploading this sensitive information to public cloud LLM endpoints (like OpenAI or Anthropic APIs) for processing is fundamentally incompatible with corporate security policies and regulatory compliance frameworks.
This necessitates the deployment of a "Sovereign Stack"—an architectural approach where the AI inference engine, vector stores, and orchestration layers run entirely within the enterprise's secure firewall or Virtual Private Cloud (VPC). The convergence of high-performance open-weights models (via Ollama) and standardized connectivity protocols (MCP) has finally made this on-premise agentic architecture viable, allowing organizations to bring the AI to the data, rather than moving the data to the AI.5
2. Theoretical Framework: The Model Context Protocol (MCP)
2.1 Decoupling Intelligence from Integration
The integration landscape for GenAI has historically been fragmented. Connecting an AI agent to a specific database or file system required developers to write bespoke connectors, managing authentication, schema discovery, and error handling for each unique data source. This "n-squared" integration problem (connecting $N$ agents to $M$ tools) created significant technical debt and vendor lock-in.
The Model Context Protocol (MCP), introduced by Anthropic and rapidly adopted as an open standard, resolves this by abstracting the connection between the AI "Host" and the Data "Server." MCP operates on a client-host-server architecture similar to the Language Server Protocol (LSP) used in IDEs.
* MCP Host: The orchestration application where the AI agent resides (e.g., a Python script using the Microsoft Agent Framework or LangGraph).
* MCP Client: A standardized library embedded within the Host that manages the protocol lifecycle, capability negotiation, and message routing.
* MCP Server: A standalone service that exposes specific data or functionality (Tools, Resources, Prompts) through the MCP standard.
In a PLM migration context, this decoupling is revolutionary. Instead of hardcoding logic to connect to an Oracle database, the migration agent simply connects to an "Oracle MCP Server." If the underlying database changes to PostgreSQL, the agent's logic remains untouched; only the MCP Server is swapped. This modularity transforms the migration architecture from a monolithic script into a composable ecosystem of specialized services.7
2.2 The Architecture of Connectivity: Transport Layers
MCP defines two primary transport mechanisms, each serving distinct needs within the migration architecture:
1. Stdio Transport: This mechanism allows the MCP Host to spawn the MCP Server as a subprocess, communicating directly via standard input/output streams. This is the preferred method for the "Sovereign Stack" in PLM migration. It ensures that the connection is strictly local and ephemeral. No network ports are opened, and data never traverses the internal network stack, providing an air-gapped environment suitable for processing highly classified engineering data.
2. Streamable HTTP (optionally using Server-Sent Events): For distributed architectures where the PLM data resides on a mainframe or a distinct secure enclave, MCP supports a Streamable HTTP transport. Clients send JSON-RPC messages via HTTP POST, and servers can optionally use Server-Sent Events (SSE) to stream multiple server-to-client messages for long-running operations and notifications. This enables the Agent to run on high-performance GPU clusters while interacting with data sources located elsewhere in the enterprise network, protected by standard HTTP authentication mechanisms (e.g., bearer tokens / API keys / OAuth).9

Important implementation notes (especially relevant for Stdio transport):
* MCP servers MUST NOT write anything to stdout that is not a valid MCP JSON-RPC message. Use stderr for logs.
* For HTTP deployments, implement standard web security protections (e.g., validate the Origin header to mitigate DNS rebinding) and prefer binding to localhost for local development.
2.3 Capability Negotiation and Tool Abstraction
A core strength of MCP is "Capability Negotiation." Upon connection, the Client and Server perform a handshake to declare their supported features. An MCP Server wrapping a legacy PLM system might declare support for "Resources" (read-only access to documents) and "Tools" (executable functions like get_bill_of_materials). This introspection allows the AI agent to dynamically discover the capabilities of the system it is migrating.
The agent does not need to know the internal schema of the legacy system beforehand. It can use MCP "Resources" to inspect the database metadata, understand the table structure, and then formulate a migration strategy. This dynamic discovery capability allows the agent to adapt to the idiosyncratic schemas of customized legacy environments without requiring hardcoded schema maps.7
3. Computational Engine: Ollama and Local Inference
3.1 The Engine of the Sovereign Stack
Ollama acts as the computational heart of the agentic migration factory. By providing a streamlined runtime for open-weights Large Language Models (LLMs) such as Llama 3, Mistral, and Phi-4, Ollama enables enterprise-grade inference on local hardware. This eliminates the latency and privacy risks associated with cloud APIs. In a migration workflow processing millions of records, the per-token cost of cloud models would be prohibitive. Ollama allows for predictable, flat-cost scaling based on hardware investment.5
The architecture leverages model quantization—reducing the precision of model weights from 16-bit to 4-bit or 8-bit—to run powerful models on standard enterprise workstations or edge servers. This means a migration agent can be deployed on a laptop connected directly to a factory floor server, extracting and processing data with the full reasoning capabilities of a 70-billion parameter model, without a single byte leaving the premises.13
3.2 Structured Outputs: Taming the Stochastic Stochasticity
A critical barrier to using LLMs in data engineering has been their non-deterministic nature. An ETL pipeline requires precise, machine-readable output; a JSON object missing a closing brace or a hallucinated key can cause the entire pipeline to fail. Ollama addresses this through native support for Structured Outputs (JSON mode).
By integrating with libraries like Pydantic, developers can define rigid schemas that the model must adhere to. When the "Mapping Agent" is asked to analyze a source column and propose a transformation, the system passes a JSON schema to Ollama defining the expected output format (e.g., fields for source_column, target_column, confidence_score, and transformation_logic). Ollama uses constrained decoding techniques to ensure that the generated tokens conform strictly to this schema. This guarantees that the agent's output can be directly parsed and executed by downstream ETL code, bridging the gap between probabilistic AI reasoning and deterministic data engineering.15
3.3 Ollama as an MCP Server
The architecture allows for a recursive pattern where Ollama itself is wrapped as an MCP Server. This enables the "Orchestrator Agent" to dynamically spawn specialized sub-agents. For example, if the migration workflow encounters a set of German-language technical specifications, the Orchestrator can call the Ollama MCP Server to spin up a specialized translation model. If it encounters complex C++ code in a legacy stored procedure, it can swap to a "CodeLlama" model to analyze the logic. This flexibility allows the migration factory to utilize the "right model for the right task," optimizing both performance and accuracy.5
4. Strategic Data Movement: From ETL to Agentic ELT
4.1 The Evolution of Data Integration
The methodology of data movement is critical. Traditional ETL (Extract-Transform-Load) processes transform data in-flight before loading it into the target. This approach is rigid; transformations are hard-coded, and if the logic is flawed, the data must be re-extracted from the source, causing downtime and performance impact. As data volumes grew, the industry shifted to ELT (Extract-Load-Transform), where raw data is loaded into a data lake or warehouse and transformed in-place using the target system's compute power. This offered greater flexibility but still relied on static SQL logic.19
4.2 The Rise of Agentic ELT
We propose a further evolution to Agentic ELT. In this paradigm, AI agents orchestrate the ELT process, introducing a layer of cognitive reasoning to the transformation phase.
1. Extract: Agents utilize Source MCP Servers to extract data. Unlike "dumb" pipes, these agents perform "intelligent extraction," pulling related clusters of data (e.g., a product and its entire revision history) to preserve locality.
2. Load: The raw data is loaded into an intermediate staging area—specifically, a Knowledge Graph (discussed in Section 5). This differs from a standard data lake by preserving the structure of the raw data.
3. Transform: This is where Agentic AI shines. Instead of running a static script, the "Transformation Agent" inspects the raw nodes in the graph. It uses LLM reasoning to identify semantic matches, resolve entities (e.g., realizing that "Vendor X" and "Vndr X" are the same), and apply normalization rules. The agent generates the transformation code (e.g., Cypher or Python) dynamically based on the data it sees.21
4.3 Self-Healing Pipelines
A key characteristic of Agentic ELT is its self-healing capability. In a traditional pipeline, a schema mismatch (e.g., a string appearing in an integer column) causes the job to fail. In an Agentic pipeline, the "Error Handler Agent" catches this exception. It analyzes the bad record, consults the LLM (via Ollama) to determine a likely fix (e.g., "The string 'N/A' in the Weight column should be treated as Null"), applies the fix, and retries the operation. This loop allows the migration to proceed autonomously through minor data quality issues that would halt a traditional process.22
5. The Context Engine: Knowledge Graphs and Data Quality
5.1 The Contextual Imperative in PLM
In PLM migration, data quality is not just about clean fields; it is about valid context. A part number is meaningless without its version history, its parent assembly, and its bill of materials. Flattening this data into relational staging tables often destroys this context, leading to "orphan" records and broken links in the target system. To maintain the Digital Thread, the migration architecture must use a storage medium that treats relationships as first-class citizens.1
5.2 Neo4j as the Migration Staging Area
We utilize Neo4j, a graph database, as the intermediate staging environment. When the "Extract Agent" pulls data from the legacy system, it writes it into Neo4j as nodes and relationships.
* Nodes: Represent entities like Part, Document, ChangeOrder, User.
* Relationships: Represent connections like CONSUMES, APPROVES, REVISES, ATTACHED_TO.
This graph-based staging allows the AI agents to perform GraphRAG (Retrieval-Augmented Generation). When the Transformation Agent needs to map a specific part, it retrieves not just the part's attributes, but its entire "neighborhood" in the graph. This allows it to make context-aware decisions. For instance, if a part lacks a "Category" attribute, the agent can look at the connected "Parent Assembly." If the parent is a "Hydraulic System," the agent can infer that the orphan part is likely a "Hydraulic Component," significantly improving classification accuracy.26
5.3 Semantic Data Profiling and Quality Analysis
The "Data Analyst Agent" leverages the graph structure to perform deep profiling that goes beyond simple column statistics.
* Structural Anomaly Detection: The agent can identify nodes that deviate from the standard connectivity pattern (e.g., a "Release" node that has no approvers).
* Semantic Clustering: By analyzing the graph topology, the agent can identify duplicate entities that have different names but identical structural roles (e.g., two "Supplier" nodes that supply the exact same set of parts).
* Cycle Detection: Circular dependencies in BOMs are a common plague in legacy data. The graph database can detect these cycles natively, allowing the agent to flag them for human review before they corrupt the target system.29
6. The Human Interface: Visual Automapping with jQuery QueryBuilder & GraphQL
6.1 Democratizing Complexity with jQuery QueryBuilder
While the backend operations are handled by sophisticated AI agents, the definition of business rules often requires the input of subject matter experts (SMEs)—engineers and PLM administrators who may not be proficient in coding. To bridge this gap, the architecture incorporates a user-friendly frontend based on jQuery QueryBuilder. This robust, open-source library provides a visual, drag-and-drop interface for constructing complex logical propositions (e.g., "IF Material is Steel AND Weight > 100 THEN Set Category to Heavy").31
In our architecture, QueryBuilder is adapted for Data Mapping and Transformation:
* Rule Definition: Users can visually construct rules to filter data for migration or define transformation logic.
* Plugin Architecture: We utilize the plugin capability to inject custom widgets. For example, a "Source Field" dropdown can be dynamically populated by the Data Analyst Agent, which has already scanned the legacy schema. This ensures that users are mapping against valid, discovered fields.32
6.2 The Visual-to-Code Transpiler Agent
The most innovative aspect of this UI integration is the use of the Mapping Agent (Ollama) as a transpiler. jQuery QueryBuilder outputs a structured JSON object representing the visual rule tree.
* Input (JSON Rule):
JSON
{
 "condition": "AND",
 "rules":
}

* Agent Task: The Orchestrator passes this JSON to the Mapping Agent with a prompt: "Convert this logic into a GraphQL query filter for the target API."
* Output (GraphQL):
GraphQL
query {
 items(where: { status: {_eq: "Released"}, created_date: {_lt: "2020-01-01"} }) {
   id
   revision
 }
}

This "AI Compiler" approach allows users to define logic in a simple UI while generating complex, syntactically correct code for the backend, handling date formats, type conversions, and nested logic automatically.34
6.3 GraphQL: The Universal Migration API
GraphQL is chosen as the interface for the Target System because of its strongly typed schema and ability to handle complex, nested data structures in a single request. Unlike REST, which might require dozens of API calls to reconstruct a product hierarchy, GraphQL allows the Migration Agent to push a Part, its Attributes, and its Relationships in a single Mutation.
The Mapping Agent utilizes GraphQL's Introspection system (__schema) to learn the target data model. It downloads the schema definition from the Target MCP Server and uses it to validate its own mapping proposals. If the target field weight is defined as a Float, the agent knows it must cast the source string "10.5 kg" to the number 10.5 before attempting the mutation. This "Type-Aware" migration significantly reduces load errors.37
7. Implementation Architecture: The Agentic Migration Factory
7.1 Agent Personas and Collaboration
The architecture is realized through a Multi-Agent System (MAS), typically orchestrated by a framework like Microsoft Agent Framework or LangGraph. We define four specialized agent personas, each with distinct responsibilities and toolsets.
Agent Persona
	Role Description
	MCP Tools & Resources
	Orchestrator Agent (User Proxy)
	Manages the high-level workflow, decomposes the user's migration goal into sub-tasks, handles error escalation, and manages the Human-in-the-Loop approval process.
	Agent Registry, Notification Tool, Workflow State Manager
	Data Analyst Agent
	Responsible for "Discovery." It connects to the Source MCP Server, profiles the legacy data, identifies schema patterns, detects data quality anomalies (nulls, duplicates), and builds the "Schema Graph" in Neo4j.
	SQL Execution Tool, Graph Query Tool, Data Profiling Tool
	Mapping Agent
	The "Reasoning Engine." It compares the Source Schema Graph with the Target GraphQL Schema. Using semantic similarity (vector embeddings of field names/descriptions), it generates mapping proposals and transformation code (Python/Cypher).
	Vector Database (Schema Store), LLM Reasoning Tool, Code Generator
	Quality Monitor Agent
	The "Validator." It takes the code generated by the Mapping Agent and runs it against a sample dataset in the staging graph. It calculates validity scores (e.g., "98% compliance") and flags violations for review.
	Validation Tool, Reporting Tool, Test Runner
	7.2 The Workflow Lifecycle
   1. Initialization: The user defines the scope (e.g., "Migrate the Engine Assembly project") via the web UI.
   2. Discovery: The Data Analyst Agent scans the source tables related to the scope. It identifies that the ENG_PART table has a foreign key to ENG_MAT. It creates a representation of this schema in the Neo4j Knowledge Graph.
   3. Proposal: The Mapping Agent analyzes the Neo4j schema. It sees ENG_MAT and looks at the Target GraphQL Schema. It finds a type MaterialSpecification. It proposes mapping ENG_MAT.chem_comp to MaterialSpecification.composition.
   4. Human Review: This proposal is pushed to the UI. The jQuery QueryBuilder is pre-populated with the rule Map Source.chem_comp -> Target.composition. The user reviews and approves.
   5. Code Generation: The Mapping Agent receives the approval. It uses Ollama to generate a Python script that implements this transformation, handling edge cases (e.g., null values).
   6. Simulation: The Quality Monitor Agent runs the script on a 10% data sample in the staging graph. It reports: "Success rate 99.5%. 5 records failed due to invalid characters."
   7. Refinement & Execution: The Mapping Agent adjusts the script to sanitize the invalid characters. The Orchestrator then approves the full execution. The data is transformed in Neo4j and then pushed to the Target System via bulk GraphQL mutations.11
7.3 Handling Edge Cases and Self-Correction
The system includes a robust "Exception Handling Loop." If a record fails to load into the target (e.g., due to a unique constraint violation on Part_Number), the error is captured by the MCP Client. The Orchestrator creates a "Repair Ticket" and assigns it to the Mapping Agent. The agent analyzes the specific error message and the data record. It might reason: "The Part Number already exists. This is likely a revision. I will append '_REV_B' to the ID and retry." It attempts the fix. If successful, it logs the remediation; if it fails again, it escalates to the human user. This autonomous error recovery dramatically reduces the manual "babysitting" required for large migrations.21
8. Operational Governance: Human-in-the-Loop
8.1 The "User Proxy" as Guardian
While autonomy is the goal, unchecked AI action on critical IP is a risk. The Orchestrator Agent acts as a "User Proxy," configured with strict governance rules. In the Microsoft Agent Framework, this agent can be set to human_input_mode="CRITICAL". This means that for read-only operations (Discovery), the agents run autonomously. However, for any operation that modifies data (Transform, Load) or deletes data, the agent must pause and request explicit permission.11
8.2 The Approval UI
When the system pauses for approval, it presents the user with a concise summary in the UI:
   * "The Mapping Agent proposes to merge Table A and Table B based on 95% data overlap."
   * "Impact: 50,000 records will be modified."
   * "Sample: -> [After] view of 5 records."
The user can click "Approve," "Reject," or "Edit Logic." Editing the logic opens the jQuery QueryBuilder, allowing the user to tweak the rule before sending it back to the agent. This workflow ensures that the human expert remains the ultimate arbiter of truth while the AI handles the execution details.42
9. Conclusion and Future Outlook
The architecture presented in this report represents a fundamental shift in how enterprises approach PLM data migration. By moving from static scripts to dynamic, context-aware Agentic AI, organizations can overcome the rigidity and "context gap" that plague traditional projects.
The combination of technologies is synergistic:
      * MCP provides the modular connectivity, allowing the system to adapt to any source/target environment without rewriting the intelligence.
      * Ollama and Local Models ensure that this powerful intelligence remains "Sovereign," protecting the organization's most valuable IP from external exposure.
      * Knowledge Graphs serve as the memory of the migration, preserving the critical relationships that define the digital thread.
      * Agentic ELT allows for iterative, non-destructive data transformation.
      * jQuery QueryBuilder & GraphQL provide the necessary bridge between the AI's reasoning and the human expert's intent.
As these technologies mature, we anticipate a future where migration is not a "one-off event" but a continuous process. "Data Steward Agents" will run perpetually in the background, monitoring data quality, harmonizing schemas across connected systems, and ensuring that the digital thread remains unbroken as the enterprise evolves. This architecture lays the foundation for that Autonomous Enterprise.

   11. GoodpointAI Repository Alignment Notes (Implementation Snapshot)
   This document describes a reference architecture. In this GoodpointAI repository, the current implementation already includes:
   * A React/Vite frontend (`agentic-restored/e2etraceapp`) with an Admin Config Manager UI for configuring LLM providers, embedding models, and connections/data sources.
   * A FastAPI backend (`agentic-restored/python_backend`) that persists these configurations and provides test endpoints, plus a Migration Wizard that consumes data sources.
   * Developer tasks in `.vscode/tasks.json` for starting frontend/backend, running frontend lint/tests, and launching the MCP Inspector.
   * A backend dependency entry for the MCP Python SDK (`mcp[cli]`) in `agentic-restored/python_backend/requirements_external_integrations.txt` to support implementing MCP servers/clients in Python.

   Key gaps to close to fully realize the MCP-based migration factory described above:
   * Implement one or more MCP Servers (stdio for local/sovereign mode and/or Streamable HTTP for remote mode) that expose migration-relevant Tools/Resources/Prompts.
   * Add a graph staging layer (Neo4j or equivalent) and an ingestion/profiling pipeline that builds a schema/context graph.
   * Implement the visual automapping and rule transpilation workflow (QueryBuilder/JSON rules -> validated transformation code / GraphQL filters/mutations) integrated into the existing UI.
   * Add explicit human-in-the-loop approvals and audit logging around any write/delete operations executed by agents.
